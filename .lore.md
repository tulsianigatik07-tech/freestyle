### Preference

<!-- lore:019f2164-6fb0-7f5a-ac0b-4b58eddf7dc6 -->
* **Always annotate key behavioral invariants with 'never' statements tied to specific file locations**: When exploring a codebase, the user explicitly calls out hard behavioral guarantees using 'never' phrasing (e.g., 'never executes plugin code', 'never filtered by disabled lists', 'never crashes plugin discovery', 'never leaves a half-written package'). These are always tied to a specific file and line number. When summarizing code behavior, mirror this pattern: extract and surface absolute invariants as 'never' assertions with precise file:line citations, not just general descriptions. These invariants are treated as load-bearing facts worth highlighting separately from ordinary behavior descriptions.

<!-- lore:019f2446-e58d-73a1-9d6c-7943d4c6dc9a -->
* **Always apply changes locally first before committing for preview**: The user consistently wants code changes applied to local files first so they can preview the diff before any commit is made. Do not commit or push changes automatically after completing edits. Leave changes uncommitted and summarize what was changed. Only commit when the user explicitly requests it. This applies to all types of changes — feature additions, removals, refactors, and fixes. After completing changes, always ask 'Want me to build and commit?' rather than committing automatically. Confirmed again: all tone.tsx UI changes in July 4 session remained uncommitted at session end.

<!-- lore:019f18ae-cee7-715a-992d-cc46ca53e68a -->
* **Always assert design intent via code comments as authoritative constraints**: Treat code comments as authoritative design constraints during review: When code comments state design intent (e.g., 'The user never chooses the model', 'Background model setup — never a decision'), treat them as binding constraints that override PR suggestions. If a PR deviates from a pattern documented in a code comment, flag it as a regression against stated intent, not a neutral trade-off. Similarly, treat verbal assertions of behavioral rules as equivalent to documented design requirements.

<!-- lore:019f14f3-5fcd-7734-870a-1d873352caef -->
* **Always batch multiple GitHub issues into a single PR creation request**: When working on GitHub issues, the user groups multiple related issues together and asks the assistant to create PRs for all of them in one request (e.g., 'create PRs resolving issues #201, #202, #318'). Prior to requesting PRs, the user typically conducts thorough read-only investigations of each issue. The assistant should handle all requested issues in the same session, creating separate PRs per issue (or combined where appropriate), without requiring the user to repeat the request issue-by-issue.

<!-- lore:019f2bdb-5854-75ae-9ad8-be108417392d -->
* **Always batch multiple unrelated tasks into a single message**: The user consistently sends multiple distinct, unrelated tasks in a single message rather than one at a time.
